Section 01

Core Python — the language itself

If you fumble these, the rest of the interview becomes uphill. For a senior role they aren't checking whether you know the answer — they're checking whether you can explain why Python made that choice and where the trade-off bites.

Easy Mutable vs Immutable — what are they, and why does it matter?

Immutable: int, float, bool, str, tuple, frozenset, bytes. Once created, the object can't change; rebinding the name creates a new object.

Mutable: list, dict, set, bytearray, and most user-defined classes.

Why it matters in interviews:

  • Mutable default arguments are a classic bug.
  • Immutables are hashable → can be dict keys / set members.
  • Strings are immutable — repeated concatenation in a loop is O(n²); use "".join().
# The famous trap
def append(x, items=[]):   # default evaluated ONCE at def time
    items.append(x)
    return items

append(1)   # [1]
append(2)   # [1, 2]  ← shared list!

# Fix:
def append(x, items=None):
    if items is None:
        items = []
    items.append(x)
    return items
Senior-level follow-up: "Is a tuple always immutable?" Trick — the tuple itself is, but if it holds a list, that list is still mutable. t = (1, [2]); t[1].append(3) works.
Easy is vs ==

== calls __eq__ — value equality. is compares identity (same object in memory, i.e. id(a) == id(b)).

Use is only for None, True, False, and sentinel checks.

a = 256; b = 256; a is b   # True — small int caching (-5..256)
a = 257; b = 257; a is b   # False (CPython detail, don't rely on it)
Medium Explain the GIL. When does it hurt, when doesn't it?

The Global Interpreter Lock is a mutex in CPython that allows only one thread to execute Python bytecode at a time. It exists because CPython's memory management (reference counting) isn't thread-safe without it.

Hurts: CPU-bound workloads — pure-Python number crunching across multiple threads doesn't scale on multi-core.

Doesn't hurt:

  • I/O-bound work (network, disk) — the GIL is released during I/O syscalls, so threads still help.
  • C extensions like NumPy that release the GIL during heavy compute.
  • multiprocessing — separate processes, separate GILs.

2026 update worth mentioning: PEP 703 — free-threaded CPython (no-GIL build) is available experimentally in 3.13+. Cite it to show currency.

Medium Shallow vs deep copy 
import copy
a = [[1, 2], [3, 4]]
b = copy.copy(a)       # new outer list, same inner lists
c = copy.deepcopy(a)   # new everything

a[0].append(99)
print(b)   # [[1, 2, 99], [3, 4]]   ← shallow!
print(c)   # [[1, 2], [3, 4]]

Deep-copy uses a memo dict to handle cyclic references and to avoid duplicating objects encountered twice.

Medium What does __slots__ do?

It tells Python to allocate fixed storage for a known set of attributes instead of a per-instance __dict__. Result: lower memory (often 40–50% per instance), faster attribute access, but no dynamic attribute addition, no __weakref__ by default, and gotchas with multiple inheritance.

class Point:
    __slots__ = ('x', 'y')
    def __init__(self, x, y):
        self.x, self.y = x, y

p = Point(1, 2)
p.z = 3   # AttributeError

Use it for value objects you'll create millions of (analytics, simulations, geometric primitives).

Medium Truthiness — what evaluates to False?

None, False, 0, 0.0, 0j, empty containers ("", [], {}, (), set()), and objects whose __bool__ or __len__ returns False/0.

Gotcha: if x: and if x is not None: are NOT equivalent — the first is False for 0 and "" too.
Hard What happens when you do import foo?
  1. Check sys.modules — if already loaded, return that.
  2. Walk finders in sys.meta_path to locate the module.
  3. The finder returns a loader; the loader creates a module object, registers it in sys.modules, then executes the module code in that module's namespace.
  4. Bind the name foo in the current scope.

Knowing this explains: circular imports (the half-built module is already in sys.modules), why importlib.reload is tricky, and how namespace packages work.

Hard LEGB scope rule + global / nonlocal

Name resolution: Local → Enclosing → Global → Built-in.

x = "global"
def outer():
    x = "enclosing"
    def inner():
        nonlocal x    # modifies outer's x
        x = "modified"
    inner()
    print(x)        # "modified"

Without nonlocal, assignment creates a new local binding — a common closure bug source.

Hard Python 3.10+ structural pattern matching — when actually useful?

It's not just switch. It destructures.

def handle(event):
    match event:
        case {"type": "click", "target": target, "shift": True}:
            return shift_click(target)
        case {"type": "click", "target": target}:
            return click(target)
        case {"type": "key", "code": code} if code.isdigit():
            return handle_digit(code)
        case _:
            return None

Real win is in parsers, AST walkers, event dispatchers, and protocol message handlers.

Senior Name 5 features you actually use from modern Python (3.10–3.13).
  • Structural pattern matching (3.10) — see above.
  • Better error messages — fine-grained SyntaxError locations, "did you mean…" suggestions.
  • Type generics syntax (3.12) — def first[T](xs: list[T]) -> T: ... without TypeVar.
  • Exception groups & except* (3.11) — needed for asyncio.TaskGroup.
  • tomllib in stdlib (3.11) for parsing pyproject.toml without external deps.
  • Free-threaded build (PEP 703) — experimental no-GIL (3.13+).
  • JIT compiler (PEP 744) — copy-and-patch JIT in 3.13.

Easy When to use list vs tuple?

Use a tuple when the collection's identity = its contents (records, return values, dict keys), or when the data shouldn't change. Use a list when the collection grows, shrinks, or gets sorted in place.

Performance: tuples are slightly smaller and faster to create — but don't over-optimize on that alone.

Medium How does a Python dict actually work?

CPython dicts are open-addressed hash tables. Since 3.6, they use a compact, ordered layout: a sparse array of indices points into a dense array of (hash, key, value) entries. That's why insertion order is preserved without extra cost.

Collisions are resolved with perturbation probing (not linear probing). Load factor stays under ~⅔; on overflow the table is rebuilt at the next power of 2.

For the interview: mention amortized O(1), the worst-case O(n) under adversarial inputs (mitigated by hash randomization with PYTHONHASHSEED for strings/bytes), and that keys must be hashable + their hash stable for their lifetime in the dict.

Medium collections module — the four classes you should name without hesitating
  • defaultdict(factory) — auto-creates missing keys.
  • Counter — multiset, with .most_common(k), addition/subtraction, and elementwise min/max.
  • deque — O(1) appendleft/popleft, maxlen for sliding windows.
  • ChainMap — layered lookups (config defaults → env → CLI).
  • (OrderedDict is mostly historical; use plain dict, but its move_to_end is still useful — e.g. LRU caches.)
Hard Implement an LRU cache from scratch

The interview classic. The pattern: hash map for O(1) lookup + doubly-linked list for O(1) eviction. In Python you can fake it with OrderedDict:

from collections import OrderedDict

class LRUCache:
    def __init__(self, capacity: int):
        self.cap = capacity
        self.d = OrderedDict()

    def get(self, key):
        if key not in self.d: return -1
        self.d.move_to_end(key)
        return self.d[key]

    def put(self, key, value):
        if key in self.d:
            self.d.move_to_end(key)
        self.d[key] = value
        if len(self.d) > self.cap:
            self.d.popitem(last=False)   # pop oldest

If they want it "from real scratch", build a Node(prev, next, key, value) doubly-linked list plus a dict key → Node. Mention thread-safety and ask whether they want it (a lock around get/put, or use functools.lru_cache which is already thread-safe).

Hard Why don't sets preserve insertion order, but dicts do?

Dicts got the compact-ordered layout in 3.6 and the language guarantees order from 3.7. Sets weren't given that guarantee — they share the hash-table machinery but operate on a sparse array of (hash, key) only, and CPython hasn't paid the engineering cost to also make them compact-ordered. If you need ordered uniqueness, use dict.fromkeys(seq).

Hard heapq — implement top-K largest 
import heapq
def top_k(stream, k):
    h = []
    for x in stream:
        if len(h) < k:
            heapq.heappush(h, x)
        elif x > h[0]:
            heapq.heapreplace(h, x)
    return sorted(h, reverse=True)

Time O(n log k), space O(k). Note heapq is a min-heap — negate values or store tuples (-priority, item) for max-heap behavior.

Section 03

OOP — beyond "explain polymorphism"

For a 7-year role they'll go straight past textbook definitions. Expect MRO, metaclasses, descriptors, dataclasses, ABCs and Protocols.

Easy Four pillars — give a Python example, not a definition
  • Encapsulation: _protected, __private (name-mangled to _Class__attr), @property.
  • Inheritance: class Child(Parent): — but prefer composition.
  • Polymorphism: duck typing — "if it quacks, it's a duck." Two unrelated classes both implementing __iter__ are interchangeable in for loops.
  • Abstraction: abc.ABC or typing.Protocol for structural typing.
Medium Explain MRO and the C3 linearization

Method Resolution Order is the deterministic sequence Python walks when looking up an attribute on a class with multiple parents. CPython uses C3 linearization: it preserves (1) the order parents are declared and (2) each class's own MRO, and disallows ambiguity.

class A: pass
class B(A): pass
class C(A): pass
class D(B, C): pass

D.__mro__   # (D, B, C, A, object)

If C3 can't produce a consistent order, Python raises TypeError at class creation. super() walks the MRO, which is why "cooperative multiple inheritance" works only when every class in the chain calls super().__init__().

Medium @classmethod vs @staticmethod — and when to actually use each

classmethod: receives the class as cls. Use it for alternative constructors:

class Date:
    def __init__(self, y, m, d): self.y, self.m, self.d = y, m, d
    @classmethod
    def from_iso(cls, s):
        y, m, d = map(int, s.split("-"))
        return cls(y, m, d)   # cls so subclasses get the right type

staticmethod: no class, no instance — just a function that logically belongs in the class namespace. Use sparingly; usually a module-level function is fine.

Medium dataclass vs NamedTuple vs Pydantic
  • NamedTuple: immutable, tuple-compatible, lightweight, hashable. Good for return values and lookup keys.
  • @dataclass: standard library; gives you __init__, __repr__, __eq__; mutable by default, or frozen=True; supports defaults via field(default_factory=list).
  • Pydantic BaseModel: runtime type validation, JSON (de)serialization, custom validators. The right tool for API boundaries, config, and untrusted input.
from dataclasses import dataclass, field
@dataclass(slots=True, frozen=True)
class User:
    id: int
    name: str
    tags: list[str] = field(default_factory=list)
Hard Descriptors — what are they and where do you actually meet them?

A descriptor is any object that defines __get__, __set__, or __delete__. When you access an attribute on an instance, Python checks the class for a descriptor and calls its protocol methods.

You use descriptors every day without realizing: @property, @classmethod, @staticmethod, Django/SQLAlchemy ORM fields, functools.cached_property.

class PositiveInt:
    def __set_name__(self, owner, name): self._name = "_" + name
    def __get__(self, obj, objtype=None):
        return getattr(obj, self._name)
    def __set__(self, obj, value):
        if value < 0: raise ValueError("must be ≥ 0")
        setattr(obj, self._name, value)

class Account:
    balance = PositiveInt()
    def __init__(self, b): self.balance = b
Hard ABC vs Protocol — pick one

abc.ABC: nominal typing — class has to explicitly inherit to count. Best when you want to enforce a contract and there's a real "is-a" relationship.

typing.Protocol: structural typing — duck typing for the type checker. A class is "a Protocol" simply by implementing the methods. Best when you want to type third-party or unrelated classes that happen to share an interface.

from typing import Protocol
class SupportsClose(Protocol):
    def close(self) -> None: ...

def shut(x: SupportsClose) -> None:
    x.close()   # works on files, sockets, your own classes — no inheritance needed
Senior Metaclasses — when have you actually used one?

A metaclass is the class of a class. type is the default. Custom metaclasses let you intercept class creation — modify attributes, register subclasses, enforce naming, etc.

The honest senior answer: "Rarely. 9 times out of 10, __init_subclass__ or class decorators are simpler and sufficient. Real metaclass cases I've seen: ORM field collection (Django Model, SQLAlchemy declarative base), Pydantic v1, abstract base class machinery."

class Registry(type):
    registry = {}
    def __new__(mcs, name, bases, ns):
        cls = super().__new__(mcs, name, bases, ns)
        Registry.registry[name] = cls
        return cls

class Plugin(metaclass=Registry): pass
class Auth(Plugin): pass   # auto-registered

Bring up "prefer __init_subclass__" — it's the modern, simpler answer for subclass tracking.

Section 04

Iterators, generators, functional bits

If someone asks "what's the difference between an iterable and an iterator", be ready to draw the protocol on the whiteboard.

Easy Iterable vs iterator vs generator
  • Iterable: any object with __iter__ returning an iterator. Lists, tuples, dicts, files, strings.
  • Iterator: has __next__ AND __iter__ (returning self). Single-pass — once exhausted, that's it.
  • Generator: a function with yield. Implements the iterator protocol automatically and lazily produces values.
def fib():
    a, b = 0, 1
    while True:
        yield a
        a, b = b, a + b

g = fib()
[next(g) for _ in range(7)]   # [0, 1, 1, 2, 3, 5, 8]
Medium When would you choose a generator over returning a list?
  • The data is huge or infinite (log files, sensor streams).
  • The consumer may stop early — don't compute what you don't need.
  • You want to pipeline transformations without intermediate lists.
  • Memory is constrained.

Trade-off: generators are single-pass and not random-access — no len(), no slicing. If the consumer needs the data twice, materialize once into a list.

Medium What's yield from for?

Delegates iteration to a sub-generator and (importantly) passes send(), throw(), and the final return value through. The pre-asyncio basis for coroutines.

def flatten(nested):
    for item in nested:
        if isinstance(item, list):
            yield from flatten(item)
        else:
            yield item
Medium map, filter, reduce vs comprehensions

Comprehensions are almost always more Pythonic and faster (no function-call overhead per element). Reach for map/filter only when you already have a named function — e.g. map(int, tokens).

reduce lives in functools. Use it for non-trivial accumulators (e.g. multiplying everything) but for sums, use sum(); for min/max, use the built-ins.

Hard Five itertools functions you'd reach for
  • chain(*iters) — concat without copying.
  • groupby(iter, key) — like SQL GROUP BY but only on contiguous runs; sort first.
  • islice(iter, start, stop, step) — slice a generator.
  • product / permutations / combinations — combinatorics without writing loops.
  • accumulate(iter, func) — running totals.
  • tee(iter, n) — fan out an iterator into n independent ones (careful: it buffers).
Hard Closure pitfall — what does this print? 
fns = [lambda: i for i in range(3)]
print([f() for f in fns])

Output: [2, 2, 2]. Each lambda captures the variable i by reference; by the time you call them, i == 2.

Fix: [lambda i=i: i for i in range(3)] — default arguments freeze the value at definition time.

Section 05

Decorators, context managers, the dunder layer

Senior code reviews are full of "wrap this in a decorator" and "use a context manager for cleanup." Be fluent.

Easy Write a timing decorator 
import functools, time

def timed(fn):
    @functools.wraps(fn)
    def wrapper(*args, **kwargs):
        t0 = time.perf_counter()
        result = fn(*args, **kwargs)
        print(f"{fn.__name__} took {time.perf_counter()-t0:.4f}s")
        return result
    return wrapper

The @functools.wraps bit matters — it preserves the original function's name, docstring, and signature for debuggers and introspection.

Medium Decorator with arguments 
def retry(times=3, exceptions=(Exception,)):
    def deco(fn):
        @functools.wraps(fn)
        def wrapper(*a, **kw):
            for attempt in range(times):
                try:    return fn(*a, **kw)
                except exceptions:
                    if attempt == times - 1: raise
        return wrapper
    return deco

@retry(times=5, exceptions=(ConnectionError,))
def fetch(url): ...

Three layers: factory → real decorator → wrapper. Worth practicing until you can write it in 90 seconds.

Medium Two ways to write a context manager 
# 1. Class-based
class Timer:
    def __enter__(self): self.t0 = time.perf_counter(); return self
    def __exit__(self, exc_type, exc, tb):
        self.dt = time.perf_counter() - self.t0
        return False   # don't suppress exceptions

# 2. contextlib.contextmanager
from contextlib import contextmanager
@contextmanager
def timer():
    t0 = time.perf_counter()
    try:    yield
    finally: print(time.perf_counter() - t0)

Returning True from __exit__ suppresses the exception — useful for "only-on-this-error" handlers but easy to misuse. Default to return False or no return.

Hard contextlib tools every senior should know
  • @contextmanager — generator-based.
  • ExitStack — manage a dynamic number of context managers (e.g. opening N files).
  • suppress(*exc) — replace try/except: pass.
  • closing(thing) — call .close() on exit.
  • redirect_stdout / redirect_stderr — handy in tests.
  • nullcontext — placeholder when a CM is conditional.

Medium Show a thread pool and a process pool 
from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor
import requests

urls = [...]
with ThreadPoolExecutor(max_workers=20) as ex:
    pages = list(ex.map(requests.get, urls))    # I/O-bound → threads

with ProcessPoolExecutor() as ex:
    results = list(ex.map(crunch, big_inputs))   # CPU-bound → processes

Prefer concurrent.futures over raw Thread/Process — same API for both, returns Futures with timeouts and cancellation.

Medium async/await — the mental model

An async function returns a coroutine object when called. The coroutine doesn't run until awaited or scheduled in an event loop. The loop runs one coroutine at a time on a single thread; at every await it can switch to another ready coroutine.

import asyncio, aiohttp

async def fetch(session, url):
    async with session.get(url) as r:
        return await r.text()

async def main(urls):
    async with aiohttp.ClientSession() as s:
        async with asyncio.TaskGroup() as tg:    # 3.11+
            tasks = [tg.create_task(fetch(s, u)) for u in urls]
    return [t.result() for t in tasks]

asyncio.run(main(urls))

The cardinal rule: never call blocking code inside a coroutine. If you must, wrap with await asyncio.to_thread(blocking_fn, ...).

Hard Producer-consumer with bounded queue 
import asyncio, random

async def producer(q, n):
    for i in range(n):
        await q.put(i)
        await asyncio.sleep(random.random() * 0.01)
    await q.put(None)   # sentinel

async def consumer(q):
    while True:
        item = await q.get()
        if item is None: break
        handle(item)
        q.task_done()

async def main():
    q = asyncio.Queue(maxsize=100)
    await asyncio.gather(producer(q, 1000), consumer(q))

The maxsize applies backpressure — producers block when the queue is full. Talk about that explicitly; senior interviewers love hearing the word "backpressure".

Hard What's deadlock and how do you avoid it?

Deadlock: ≥ 2 tasks each hold a resource the other needs, none release. The Dining Philosophers problem.

Avoidance strategies:

  • Lock ordering — always acquire locks in a fixed global order.
  • Timeouts — lock.acquire(timeout=N), then back off.
  • Lock-free designs — message passing via queues instead of shared mutable state.
  • Use higher-level primitives — RLock, Semaphore, Event, Condition when they fit.
Senior Compare multiprocessing.Queue vs asyncio.Queue vs queue.Queue
  • queue.Queue: thread-safe, blocking. Use between threads in the same process.
  • multiprocessing.Queue: cross-process, backed by a pipe + lock + helper thread; objects are pickled.
  • asyncio.Queue: NOT thread-safe, single-event-loop only. Methods are coroutines (await q.get()).
Trap: putting an asyncio.Queue across threads silently breaks. To bridge threads ↔ async, use loop.call_soon_threadsafe or asyncio.run_coroutine_threadsafe.
Section 07

Memory, GC, performance

"My script is slow / leaking" is a question, not a story. Be ready to talk about how you'd actually debug it.

Medium How does Python's garbage collector work?

Two mechanisms:

  1. Reference counting (primary). Every object has a refcount. When it hits 0, the object is freed immediately and its references are decremented.
  2. Generational cyclic GC (gc module). Refcounting can't free reference cycles (a.ref = b; b.ref = a). A separate collector walks objects in three generations (young / middle / old), promoting survivors, and breaks cycles.

You can disable the cyclic GC with gc.disable() in latency-sensitive paths (e.g. Instagram famously did this in early Django days), then re-enable.

Medium Tools you'd use to find a memory leak
  • tracemalloc (stdlib) — snapshot diff to find growing allocations.
  • objgraph — visualize ref graphs, find what's keeping an object alive.
  • memray (Bloomberg) — production-grade memory profiler.
  • guppy3 / pympler — heap summaries.
  • resource.getrusage or /proc/self/status on Linux for RSS over time.

Common causes: globals/caches without bounds, circular references with __del__, holding references to large objects inside closures, module-level singletons.

Hard Profile CPU — what's your workflow?
  1. Measure: cProfile + snakeviz for a visual flame graph, or py-spy on production processes (sampling, no instrumentation).
  2. Find the hot path. Don't optimize without numbers.
  3. Pick the right fix: better algorithm > better data structure > vectorize with NumPy > caching > rewrite in Cython / Rust (pyo3) / C extension.
  4. Verify with the same profiler, on the same workload.

Sample answer framing

"My default is py-spy record against the running service — it's sampling-based so production overhead is <5%, and it produces flame graphs I can read in 30 seconds. For local hot loops I'll switch to line_profiler for per-line timing."

Hard Make this slow function fast 
# Before: O(n²)
def find_duplicates(items):
    dups = []
    for i in range(len(items)):
        for j in range(i+1, len(items)):
            if items[i] == items[j] and items[i] not in dups:
                dups.append(items[i])
    return dups

# After: O(n)
from collections import Counter
def find_duplicates(items):
    return [k for k, v in Counter(items).items() if v > 1]

This is a common screening question. The first version has three O(n) costs hiding: the inner loop, the in dups check (list membership), and the worst case of repeated duplicates.

Senior When would you reach outside Python for performance?
  • NumPy / Pandas / Polars — vectorize numerical work. 10×–100× wins for free.
  • Cython / mypyc — type-annotate hot Python, compile to C.
  • Numba — JIT-compile numerical functions.
  • PyO3 / maturin — Rust extension. Increasingly common in 2026 (ruff, pydantic-core, polars).
  • ctypes / cffi — call existing C libraries.

Testing, packaging, tooling

Easy pytest features that beat unittest
  • Plain assert with rich introspection.
  • Fixtures (dependency injection for tests) with scope: function/class/module/session.
  • Parametrize for table-driven tests.
  • Plugin ecosystem: pytest-xdist (parallel), pytest-cov, pytest-asyncio.
@pytest.fixture
def db():
    conn = connect()

    conn.close()

@pytest.mark.parametrize("a,b,expected", [(1,1,2), (2,3,5)])

    assert add(a, b) == expected
Medium Mock vs patch vs MagicMock — talk through it
  • Mock — bare stand-in object; attributes/calls auto-vivify.
  • MagicMock — Mock with all magic methods pre-configured (works as a context manager, iterator, etc.).
  • patch — temporarily replace an object at a path with a Mock during a test. Always patch where the name is looked up, not where it's defined.
# If module b does: from a import api
# Then patch b.api, not a.api
with patch("b.api") as m:
    m.return_value = {"ok": True}
    assert b.do_thing() == "ok"
Medium What does "good test pyramid" look like for a Python service?

  • A small number of end-to-end tests — full stack, real network. Slow but high-confidence.
  • Plus: property-based tests via hypothesis for invariants; contract tests for service boundaries.

Section 09

Django · Flask · FastAPI

If the job description names a framework, go deep on that one. If not, know FastAPI well and Django at a working level — that combo covers most 2026 listings.

Easy Django vs Flask vs FastAPI — one-line each
  • Django: batteries-included monolith (ORM, admin, auth, templates). Best for content sites and large team apps.
  • Flask: minimal WSGI micro-framework. Pick your own batteries. Great for small services and prototypes.
  • FastAPI: ASGI, async-first, Pydantic-based request validation, auto OpenAPI/Swagger. The current default for new Python APIs.
Medium WSGI vs ASGI — what's the actual difference?

WSGI: synchronous, request → response, one call per request. Servers: gunicorn, uWSGI. Used by Django (until 3.x), Flask.

ASGI: async, supports long-lived connections (WebSockets, SSE, HTTP/2). Servers: uvicorn, hypercorn, daphne. Used by FastAPI, Starlette, modern Django.

For a senior follow-up: "ASGI is a superset — you can wrap a WSGI app in an ASGI server with adapters like a2wsgi, but the inner app still blocks the event loop."

Medium Django request lifecycle
  1. WSGI/ASGI server hands raw request to Django WSGIHandler.
  2. Django constructs a HttpRequest.
  3. Each middleware's __call__ / process_request runs.
  4. URL resolver matches the path → view.
  5. View runs (often via @login_required, @require_POST, etc.).
  6. ORM hits the DB, template renders, response object created.
  7. Middlewares unwind (in reverse) with process_response.
  8. Response goes back through the server to the client.

Useful framing for "where would you add request-id tracing?" → middleware.

Hard FastAPI dependency injection — design walkthrough 
from fastapi import Depends, FastAPI, HTTPException
from sqlalchemy.orm import Session

app = FastAPI()

def get_db():
    db = SessionLocal()
    try: yield db
    finally: db.close()

def current_user(token: str = Depends(oauth2_scheme),
                 db: Session = Depends(get_db)):
    user = db.query(User).filter_by(token=token).first()
    if not user: raise HTTPException(401)
    return user

@app.get("/me")
def me(user: User = Depends(current_user)):
    return user

Dependencies are cached per request, can be sync or async, and form a graph that FastAPI resolves before calling the route. Senior-level talking points: testability (override with app.dependency_overrides), composition, request-scoped state.

Hard Authentication patterns — JWT vs session
  • Session: cookie + server-side store. Stateful, easy to revoke, requires sticky/shared storage. Best for first-party web apps.
  • JWT: signed token; verified statelessly. Easy to scale horizontally, hard to revoke before expiry (need a blocklist). Use short-lived access tokens + refresh tokens.
  • OAuth2 + OIDC: standard for third-party auth (Google, Microsoft); FastAPI has built-in helpers.
Senior Scaling a Django app to handle 10× traffic

Walk through layered options:

  1. Measure first: APM (Datadog, Sentry, NewRelic, OTel) to find the actual bottleneck.
  2. DB: add indexes, fix N+1 queries (select_related / prefetch_related), connection pooling (pgbouncer), read replicas.
  3. Caching: Redis for query cache, fragment cache, session cache.
  4. Async work: push slow tasks to Celery/RQ/Dramatiq workers.
  5. Horizontal scaling: multiple gunicorn instances behind a load balancer; stateless app servers.
  6. Static/media: CDN (CloudFront, Cloudflare) + S3.
  7. Profile hot Python paths; consider rewriting in async views or microservice.
Section 10

Databases, ORMs, the N+1 fight

Easy What's an N+1 query?

You run 1 query to fetch N parent rows, then 1 more query per row to fetch their relations — 1 + N queries instead of 1 or 2.

# Bad — 1 + N queries
for author in Author.objects.all():
    print(author.name, author.books.count())   # SQL per author

# Good — 2 queries
for author in Author.objects.prefetch_related("books"):
    print(author.name, len(author.books.all()))

Tools to detect: django-debug-toolbar, nplusone, SQLAlchemy event listeners that count queries per request.

Medium select_related vs prefetch_related
  • select_related: SQL JOIN, single query. For FK and OneToOne (forward).
  • prefetch_related: separate query, joined in Python. For ManyToMany and reverse FK.
Medium SQLAlchemy Core vs ORM

Core: expression language built on the Engine — write SQL with Python objects, get back rows. Lighter weight; full SQL control.

ORM: built on Core. Adds identity map, unit of work, relationships, lazy loading. SQLAlchemy 2.x is async-capable and has a unified select() API.

Hard Transactions — explain isolation levels

Postgres's "Serializable" uses SSI (Serializable Snapshot Isolation) and may roll back transactions with a serialization failure — your app needs retry logic.

Hard When NoSQL?
  • Document (Mongo, DynamoDB): unstructured/varied shape, low join needs, write-heavy, scale by sharding.
  • Key-value (Redis, DynamoDB): caches, session store, rate-limit counters, leaderboards.
  • Wide-column (Cassandra): massive writes, time-series, geo-distribution.
  • Graph (Neo4j): heavy relationship traversal — social, fraud, recommendations.
  • Search (Elasticsearch, OpenSearch): full-text, faceted search, log analytics.

Default: Postgres. Reach for NoSQL when you have a specific access pattern that relational handles poorly.

Senior Migrations in production — your playbook
  1. Backward-compatible first: deploy code that works against both old and new schema.
  2. Migrate: add columns/tables (additive only). Never drop or rename in one shot.
  3. Backfill: run a batched data migration in a job (LIMIT/OFFSET or keyset pagination), throttled.
  4. Switch reads/writes: deploy code that uses the new schema.
  5. Cleanup: in a later release, drop the old columns/tables.

Avoid: long-running migrations holding locks on big tables. In Postgres, use CREATE INDEX CONCURRENTLY, ADD COLUMN ... NULL without default, then backfill.

Section 11

System design — five practice rounds

At 7 YOE you'll get one or two of these. The expected answer isn't a single architecture — it's a structured conversation about requirements, trade-offs, scale, and failure modes.

Framework to use every time

1) Clarify requirements (functional + non-functional). 2) Estimate scale (QPS, storage, bandwidth). 3) API design. 4) High-level diagram. 5) Data model. 6) Deep-dive 1–2 components. 7) Bottlenecks, failure modes, scaling.

Design URL shortener (bit.ly)
  • Clarify: read/write ratio (~100:1), custom aliases, analytics, expiration?
  • Scale: 100M URLs/day → ~1.2K writes/sec, 120K reads/sec at peak.
  • Encoding: base62 of a 64-bit ID. Or hash(url)[:7] + collision check.
  • Storage: Postgres for canonical mapping; Redis for hot reads.
  • ID generation: Snowflake-style; or pre-allocated ranges per app server.
  • Edge: CDN cache 302 redirects with TTL.
  • Analytics: async write to Kafka → ClickHouse / BigQuery, never block the redirect.
Design Rate limiter

Algorithms — know all four:

  • Fixed window: count per minute. Simplest; allows bursts at boundaries.
  • Sliding window log: store timestamps; precise but memory-heavy.
  • Sliding window counter: weighted avg of two adjacent fixed windows. Best balance.
  • Token bucket: tokens refill at rate R; request costs 1 token. Allows controlled bursts.

Distributed: Redis + Lua script for atomic check-and-decrement. Or a sidecar (Envoy, nginx). Talk about clock skew, hot-key avoidance, fail-open vs fail-closed.

Design News feed (Twitter/Instagram-style)

The classic fanout-on-write vs fanout-on-read trade-off.

  • Fanout-on-write (push): when a user posts, push to all followers' feed caches. Fast reads, expensive for celebrities with 100M followers.
  • Fanout-on-read (pull): at read time, fetch latest posts from each followee. Cheap writes, slow reads.
  • Hybrid: push for normal users, pull for celebrities — Twitter's actual approach.

Cover: ranking, cache invalidation, write-heavy storage (Cassandra), media (S3 + CDN), real-time updates (WebSocket/SSE).

Design File upload + processing pipeline
  • Pre-signed S3 URL → client uploads directly, app server only mints credentials.
  • S3 event → SQS / SNS → worker pool.
  • Workers process (transcode, OCR, virus scan, thumbnail) with idempotency keys.
  • State machine: UPLOADED → PROCESSING → READY / FAILED.
  • Dead-letter queue + retry-with-backoff + alerting.
  • Notify client: WebSocket or polling endpoint.
Design Distributed job scheduler (like cron at scale)
  • Storage: DB row per job with next_run_at index.
  • Pickers: workers SELECT ... FOR UPDATE SKIP LOCKED LIMIT N — clean Postgres pattern for safe concurrent claim.
  • Execution: separate executor pool; jobs are idempotent so retries are safe.
  • Failure: exponential backoff, max retries, DLQ.
  • Observability: per-job metrics, alerting on overdue jobs.
  • Compare to: Celery beat, Airflow, Temporal, AWS Step Functions — and when you'd just use one of those instead.

DSA — the patterns that pay off

You won't be asked to invert a binary tree blindfolded, but they will probe whether you can pick the right pattern and code it cleanly.

Pattern Two pointers

For sorted arrays: pair-sum, container-with-most-water, remove-duplicates, three-sum.

def two_sum_sorted(arr, target):
    i, j = 0, len(arr) - 1
    while i < j:
        s = arr[i] + arr[j]
        if s == target: return (i, j)
        if s < target: i += 1
        else: j -= 1
Pattern Sliding window

Variable-size window; expand right, contract left while a constraint is broken.

def longest_unique(s):
    seen = {}; left = 0; best = 0
    for right, ch in enumerate(s):
        if ch in seen and seen[ch] >= left:
            left = seen[ch] + 1
        seen[ch] = right
        best = max(best, right - left + 1)
    return best
Pattern BFS / DFS on graphs and grids 
from collections import deque
def bfs(start, graph):
    seen = {start}; q = deque([start])
    while q:
        node = q.popleft()
        for nb in graph[node]:
            if nb not in seen:
                seen.add(nb); q.append(nb)
    return seen

DFS — recursion or explicit stack. For grids: 4-way directions [(-1,0),(1,0),(0,-1),(0,1)]. Track visited or mutate the grid.

Pattern Dynamic programming — the 5 standard shapes
  • 1D linear: Fibonacci, climbing stairs, house robber.
  • 2D grid: unique paths, edit distance, LCS.
  • Knapsack: 0/1, unbounded, partition.
  • Intervals: matrix chain, palindromic substrings.
  • Tree DP: post-order accumulation, e.g. max path sum.

Start with recursion + memoization; convert to tabulation only when needed for space/perf.

Pattern Backtracking template 
def backtrack(state, choices):
    if is_solution(state):
        results.append(state.copy())
        return
    for ch in choices:
        if valid(state, ch):
            state.append(ch)
            backtrack(state, choices)
            state.pop()   # undo

Combinations, permutations, N-queens, Sudoku, word search.

Behavioral & a 2-week prep plan

STAR-framed answers you should have ready

Behavioral "Tell me about a hard bug you debugged"

Structure (STAR):

  • Situation: production incident, 2 sentences of context (scale, impact).
  • Task: what your responsibility was.
  • Action: tools used (logs, py-spy, tracing), hypotheses formed, what you ruled out and why.
  • Result: root cause in one sentence, fix, MTTR, what you changed permanently (test, alert, runbook).

Make sure the bug is technically interesting. Concurrency, memory leaks, off-by-one in distributed systems, ORM N+1 hidden behind a property — all good. Avoid "I rebooted it."

Behavioral "Tell me about a technical disagreement"

The hidden test: are you reasonable, do you change your mind on evidence, do you make decisions and move on?

Structure: state both positions fairly → show you sought data (benchmark, prototype, ADR) → describe how you decided → describe how you supported the decision even if it wasn't yours.

Anti-pattern: framing yourself as obviously right and the other person as obviously wrong.

Behavioral "Tell me about mentoring or leading"

At 7 YOE they expect impact beyond your own code. Examples that work: onboarded N juniors, drove adoption of a tool/framework, ran a tech-talk series, led a migration, wrote design docs that shaped the team's direction, did interview hiring loops.
